# Архитектура подсистемы индексации документов

## Применяемые архитектурные паттерны

### 1. Shared Storage (Общее внешнее хранилище состояния)

Передача содержимого документа между сервисами осуществляется не путём вложения данных в тело сообщения, а через **общее внешнее объектное хранилище** (MinIO, совместимое с Amazon S3 API). Сервис `documents` сохраняет бинарное содержимое документа в хранилище и помещает в сообщение лишь **ссылку** на объект (`content_path`). Сервис `index-python` при получении события самостоятельно извлекает данные по этой ссылке.

Данный подход соответствует паттерну **Claim Check** (также известному как *Store-In-Library*), описанному в классификации Enterprise Integration Patterns (Hohpe, Woolf, 2003): крупный объект данных изымается из потока сообщений и помещается во внешнее хранилище, а по каналу передаётся лишь идентификатор (токен) для его последующего получения.

**Преимущества:**
- Брокер сообщений не перегружается объёмными полезными нагрузками.
- Содержимое документа доступно любому числу подписчиков без дублирования данных.
- Обеспечивается независимое масштабирование хранилища и транспортного уровня.

---

### 2. Event-Driven Architecture (Событийно-ориентированная архитектура)

Взаимодействие между сервисами построено на принципах **событийно-ориентированной архитектуры** (EDA). Сервис `documents` публикует доменное событие `corpus.document.created` в брокер сообщений NATS после успешного сохранения документа. Сервис `index-python` является независимым подписчиком этого события и реагирует на него без какой-либо прямой связи с источником.

Такая схема реализует паттерн **Publisher/Subscriber**, при котором производитель события не осведомлён о том, кто и сколько потребителей его обрабатывает. Это обеспечивает **слабую связанность** (loose coupling) компонентов системы.

---

### 3. Асинхронная обработка (Asynchronous Processing)

Публикация события в сервисе `documents` выполняется **асинхронно** в отдельной горутине с собственным контекстом и таймаутом, что исключает блокировку основного потока обработки HTTP-запроса. Сервис `index-python` реализован как постоянно работающий асинхронный процесс на базе `asyncio`, обрабатывающий входящие события по мере их поступления.

## Поток обработки нового документа

```
Клиент
  │
  ▼
[documents-service]
  1. Принимает HTTP-запрос с содержимым документа
  2. Сохраняет содержимое в MinIO → получает content_path
  3. Сохраняет метаданные документа в PostgreSQL
  4. Асинхронно публикует событие corpus.document.created в NATS
     { document_id, title, content_path, author_id, ... }
  │
  ├──► [MinIO]  ◄─────────────────────────────────────┐
  │    Хранит содержимое документа по content_path     │
  │                                                    │
  └──► [NATS]                                          │
         │  topic: corpus.document.created             │
         ▼                                             │
     [index-python]                                    │
       1. Получает событие                             │
       2. Извлекает content_path из payload            │
       3. Загружает содержимое из MinIO ──────────────►┘
       4. Разбивает текст на чанки (Chunker)
       5. Генерирует векторные представления (Ollama)
       6. Записывает векторы в Qdrant
```

## Используемые технологии

| Роль | Технология |
|---|---|
| Брокер сообщений | NATS |
| Объектное хранилище | MinIO (S3-совместимое) |
| Реляционная БД (метаданные) | PostgreSQL |
| Векторная БД (индекс) | Qdrant |
| Модель эмбеддингов | Ollama |
